你在执行 **duet 收工门禁**。目标:这段改动做到"干净 + 真验证过 + 人看过 diff + 被独立复审过"才算完。按顺序走,别跳步、别提前宣布完成。

## 1 · 收拾(乱不过夜)
自审当前 diff,按 `/simplify` 的标准过一遍:命名、结构、拆 >~300 行的文件、去过度设计和 AI 味、删掉与本次任务无关的改动。最终 diff 只留跟这次任务相关的内容。

## 2 · 验证(要真证据,不是剧场)
- 跑测试 / 跑一遍真实流程,把**关键输出原样贴出来**——不是"跑起来了""应该没问题"这种断言。
- 别只走 happy path:**主动打边界 / 空态 / 异常 / off-by-one / 罕见分支**。测试通过 ≠ 正确,尤其 AI 生成的测试常只验证了它自己的实现。
- 不能被验证的改动 = 没做完;缺验证手段就先建一个。`$ARGUMENTS` 指定的关注点一并覆盖。

## 3 · 看过 diff(YOLO 护栏 · 不能跳)
- **明确问用户:这份 diff 你亲眼看过了吗?** 没看过就先把 `git diff` 摆出来,等确认再往下走。
- 尤其:凡是 codex YOLO / rescue 直接改盘产生的改动,**默认按"未经人看过"处理**——必须先在干净分支 / worktree 里做,跑完 `git diff` 逐块看过。
- 没人看过的 diff,一律不算 ship。

## 4 · Codex 独立复审(配合 codex)
提示用户用另一个引擎独立复审(它能看到 Claude 的盲区):
- 查缺陷 / 正确性 → `/codex:review`
- 质疑做法 / 设计 / 假设 → `/codex:adversarial-review "$ARGUMENTS"`
- 改动大或不确定 → 加 `--background`,`/codex:status` 看进度、`/codex:result` 取结果
- 让 Codex **专挑**:边界条件、空态、异常处理、off-by-one、罕见路径,以及"该抽共用却复制成多份的重复实现"。

**两条红线**:
1. Codex 复审是只读的——等它出结果、按严重度列给用户、问哪些要修,**绝不自动改**。修不修由用户定。
2. **Codex 只是第一遍,不是证明**。它擅长机械缺陷,但会漏复杂逻辑 / 罕见路径,说"没问题"≠对。复杂逻辑你自己再扫一眼。

## 5 · 收尾
- 把"下一步 / 明天第一件事"写进项目的 `.duet/next.md`(一两行)——下次开会话 duet 自动顶出来,你不用记。
- **未完成的跨文件工作也写进去**:本次改了 A 但"影响到却没改"的 B/C、或临时复制没抽共用的地方,列进 `.duet/next.md` 的"未完成"区,别让它跨会话丢掉。
- **发现问题 → 固化进系统**:本轮验证 / Codex 复审若抓到"可复发类"的真问题(不是一次性笔误),别只修这一处——把教训编码进 `CLAUDE.md` / skill / lint 规则,让以后每次迭代自动受益,而不是每次重踩。
- 然后收工,别把开放循环带过夜。

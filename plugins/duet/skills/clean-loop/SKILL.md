# 干净开发循环 · clean-loop

目的:让代码"写出来就干净、每步可验",把返工和救火降到最低。累,大多来自乱代码引发的返工循环和"事情没关上",不是活本身——这个循环就是来掐断这两点。

配合 Codex:Codex(GPT-5.5)是另一个**独立**引擎,价值在于看到 Claude 自己的盲区——所以卡住时让它救场,收工前让它复审。

## 何时用 / 何时跳过
- 用:新功能、bug 修复、重构、多文件改动、不熟悉的模块。
- 跳过:一句话就能说清的改动(错别字、加一行日志、单纯重命名)。别为小事套流程,那本身就是负担。

## 四条底层原则(为什么这样不累)
1. **不硬扛**:同一处卡住约 15 分钟或两次没搞定,就交给 Codex,别 grind。
2. **小步可验**:一次只推一小片,且每片能立即验证——错了当场发现,不攒成大返工。
3. **乱不过夜**:每轮结束就收拾干净,不把乱堆到明天(本插件的 auto-tidy 钩子会自动帮你做基础格式化)。
4. **独立复审**:自己审不出自己的盲区,收工前让 Codex 过一遍。

## 循环

### 0 · 探清(Explore)——动手前先读
读清任务边界和相关代码/调用链,别拿孤立片段就改。
文件多、范围大时,把"摸清现状"派给子代理(Explore / general-purpose),主线程只留跟实现相关的结论——上下文是稀缺资源,别塞满。

### 1 · 规划结构(Plan)——定了再写
动手前先产出:
- 文件划分 / 关键命名 / 主要函数签名;
- **验证 handle**:这次"做完"长什么样(哪个测试通过 / 哪条命令跑出预期 / 什么输出)。没有验证方式,先建一个再写。

多文件或方案不确定 → 进 Plan Mode,先给计划让用户确认,再落代码。范围大时可把本步交给 `duet-planner` 子代理:独立上下文里探清 + 出方案,主线程不被探查细节撑满。一句话改动跳过本步。

### 2 · 小步实现(Implement)
- 一次一小片,每片独立可验,别写一大坨最后一起调。
- 跟现有代码风格 / 命名保持一致;KISS / DRY / YAGNI,不过度设计,不加没用的包装层。
- 卡住触发原则 1 → 走下面 Codex rescue,别硬扛。

### 3 · 每步验证(Verify)
每片写完立即验:跑测试 / 跑一遍真实流程(可用 `/verify`)。
**给出证据,不是断言**——贴测试输出 / 命令结果,而不是只说"成功了"。别只走 happy path,顺手打一下边界 / 空态 / 异常;测试过 ≠ 对(AI 测试常只验证了它自己的实现)。
不能被验证的改动,就是没做完。重要 / 复杂改动可交 `duet-verifier` 子代理独立验收——全新上下文、无偏见,专治"作者模型说自己没问题"。

### 4 · 收拾(Clean up)——乱不过夜
自审一遍,再用 `/simplify`:命名、结构、拆掉 >~300 行的文件、去过度设计和 AI 味、删掉与任务无关的改动。最终 diff 只留跟这次任务相关的内容。

### 5 · Codex 独立复审(配合 codex)
收拾完,把 diff 交给另一个引擎复审(它能看到 Claude 的盲区)。收工时可直接跑 **`/duet:ship`** 一次走完"收拾 + 验证 + 复审 + 收尾"门禁。手动方式:
- 查缺陷 / 正确性 → 提示用户跑 `/codex:review`
- 质疑"这个做法 / 设计 / 假设对不对" → `/codex:adversarial-review "<关注点>"`
- 改动大或不确定 → 加 `--background`,先去干别的,`/codex:status` 看进度、`/codex:result` 取结果

**红线**:Codex 复审是只读的。拿到结果后停下,按严重度列给用户,问哪些要修——绝不自动改。修不修由用户定。**另外:Codex 只是第一遍、不是证明**——它会漏复杂逻辑 / 罕见路径,说"没问题"≠对,复杂处自己再扫一眼。

## 执行清单(照着勾)
```
[ ] 0 探清:边界 + 相关代码读过了(文件多则派子代理)
[ ] 1 规划:结构/命名定了 + 验证 handle 明确了(多文件走 Plan Mode)
[ ] 2 实现:小步推进,风格一致,没过度设计
[ ] 3 验证:每片跑过,贴了证据
[ ] 4 收拾:/simplify 过、无关改动清掉、无 >300 行巨文件
[ ] 5 复审:/duet:ship(或 /codex:review),findings 交用户定
[ ] 收尾:记下"下一步",收工
```

## Codex 协作速查
| 场景 | 怎么调 | 说明 |
|---|---|---|
| 卡住 / 要第二实现 / 深挖根因 | `/codex:rescue "<明确任务>"` | 可写引擎;小而明确→前台,开放/多步→加 `--background` |
| 收工前一键门禁 | `/duet:ship "<关注点>"` | 收拾 + 验证 + 提示 Codex 复审 + 收尾 |
| 收工前独立查缺陷 | `/codex:review` | 只读、原样返回,不自动修 |
| 质疑设计 / 取舍 / 假设 | `/codex:adversarial-review "<关注点>"` | 挑战 approach 本身,不只挑实现瑕疵 |
| 看后台进度 / 取结果 | `/codex:status` · `/codex:result` | |
| 没装 / 没登录 | `/codex:setup` | 首次准备 Codex |

给 Codex 的 rescue 提示要点:一次一个明确任务;告诉它"做完长什么样";交代验证方式。**你的 codex 默认 YOLO(无审批、full-access、直接改盘)**,所以开跑前先切到干净分支 / worktree,泡完先看 diff 再信。

## 配套习惯(X / 社区验证,顺手就少累)
- **每个逻辑单元就 commit 一次**,别攒到收工:坏决定能单独回滚,最终 diff 干净、好复审——codex YOLO 泡完尤其要先看 diff 再信。
- **上下文到 ~60% 就 `/compact`**,别等撑满:满之前质量就会悄悄下滑,越往后越容易出错、越费你盯。
- **能写测试的地方,先写会失败的测试再实现**——"做完" = 测试变绿,把乱和 bug 挡在源头。

## 收尾
把"下一步 / 明天第一件事"写进项目的 `.duet/next.md`(一两行);**"影响到却没改的其他文件、临时复制没抽共用的地方"也一并记进"未完成"区**,别让跨文件工作跨会话丢。下次开会话 duet 的 SessionStart 钩子会自动顶出来,你不用记。然后收工,别把开放循环带过夜——累多半来自事情没关上。

# 干净开发循环 · clean-loop

目的:让代码"写出来就干净、每步可验",把返工和救火降到最低。累,大多来自乱代码引发的返工循环和"事情没关上",不是活本身——这个循环就是来掐断这两点。

配合 Codex:Codex(GPT-5.5)是另一个**独立**引擎,价值在于看到 Claude 自己的盲区——所以卡住时让它救场,收工前让它复审。

**双记忆兼容**:Claude(主攻)与 Codex(辅助)各自的跨会话记忆共存、互不干涉,duet 不做同步;规矩类内容以共享层为唯一真源(全局 CLAUDE.md+AGENTS.md 镜像、项目 `.duet/`),哪边的私有记忆和共享层冲突,以共享层为准。

## 先分诊(动态伸缩——流程按任务伸缩,不是一套套所有活)
接到任务先花 10 秒判级,按级走;拿不准就升一级:

| 级 | 特征 | 怎么走 |
|---|---|---|
| **S** | 一句话说清、单点、低风险(错别字/加日志/单纯重命名) | 直接改 + 顺手验证,不套循环(auto-tidy 兜底) |
| **M** | 单文件或小范围、目标清晰、模块熟 | 简版循环:快速探清 → 定验证 handle → 小步写 → 验证 → 收拾。主线程做完,不派子代理、不进 Plan Mode |
| **L** | 跨多文件 / 不熟模块 / 方案不确定 / 高风险 | 完整循环 0-5:duet-planner 出方案(或 Plan Mode),实现后 duet-verifier 独立验收,`/duet:ship` 收工 |
| **XL** | 大范围审计 / 迁移 / 多方案权衡,单上下文吃不下 | L 的基础上用 `/duet:swarm` 多智能体编排(并行探查 / 多验证者对抗 / 方案评审) |

**执行中动态升级**:M 干着发现跨文件 → 升 L;L 探查发现范围爆炸 → 升 XL;同一处卡 ~15 分钟或两次没搞定 → codex rescue。只升不降,别中途偷工减料。

## 四条底层原则(为什么这样不累)
1. **不硬扛**:同一处卡住约 15 分钟或两次没搞定,就交给 Codex,别 grind。
2. **小步可验**:一次只推一小片,且每片能立即验证——错了当场发现,不攒成大返工。
3. **乱不过夜**:每轮结束就收拾干净,不把乱堆到明天(本插件的 auto-tidy 钩子会自动帮你做基础格式化)。
4. **独立复审**:自己审不出自己的盲区,收工前让 Codex 过一遍。

## 循环

### 0 · 探清(Explore)——动手前先读
读清任务边界和相关代码/调用链,别拿孤立片段就改。
文件多、范围大时,把"摸清现状"派给子代理(Explore / general-purpose),主线程只留跟实现相关的结论——上下文是稀缺资源,别塞满。

### 1 · 规划结构(Plan)——定了再写
动手前先产出:
- 文件划分 / 关键命名 / 主要函数签名;
- **验证 handle**:这次"做完"长什么样(哪个测试通过 / 哪条命令跑出预期 / 什么输出)。没有验证方式,先建一个再写。

多文件或方案不确定 → 进 Plan Mode,先给计划让用户确认,再落代码。范围大时可把本步交给 `duet-planner` 子代理:独立上下文里探清 + 出方案,主线程不被探查细节撑满。一句话改动跳过本步。

### 1½ · 选实现引擎(双引擎不只是复审)
方案定了,先选谁来写,再动手:
- **Claude(默认)**:要跟现有架构 / 风格深度一致、实现中需要来回权衡取舍的活。
- **整段派给 Codex**(`/codex:rescue "<完整规格:改什么 + 做完长什么样 + 怎么验证>"`):规格清晰、机械、样板多的活,或想要一份独立第二实现来对比。**红线**:codex 是 YOLO 直接改盘——派活前先切干净分支 / worktree;泡完 Claude 逐块看 diff、派 duet-verifier 独立验收,过了才合入。
- 拿不准 → 默认分工:Claude 写、Codex 审。

### 2 · 小步实现(Implement)
- 一次一小片,每片独立可验,别写一大坨最后一起调。
- 跟现有代码风格 / 命名保持一致;KISS / DRY / YAGNI,不过度设计,不加没用的包装层。
- 卡住触发原则 1 → 走下面 Codex rescue,别硬扛。

### 3 · 每步验证(Verify)
每片写完立即验:跑测试 / 跑一遍真实流程(可用 `/verify`)。
**给出证据,不是断言**——贴测试输出 / 命令结果,而不是只说"成功了"。别只走 happy path,顺手打一下边界 / 空态 / 异常;测试过 ≠ 对(AI 测试常只验证了它自己的实现)。
不能被验证的改动,就是没做完。重要 / 复杂改动可交 `duet-verifier` 子代理独立验收——全新上下文、无偏见,专治"作者模型说自己没问题"。

### 4 · 收拾(Clean up)——乱不过夜
自审一遍,再用 `/simplify`:命名、结构、拆掉 >~300 行的文件、去过度设计和 AI 味、删掉与任务无关的改动。最终 diff 只留跟这次任务相关的内容。

### 5 · Codex 独立复审(配合 codex)
收拾完,把 diff 交给另一个引擎复审(它能看到 Claude 的盲区)。收工时可直接跑 **`/duet:ship`** 一次走完"收拾 + 验证 + 复审 + 收尾"门禁。手动方式:
- 查缺陷 / 正确性 → 提示用户跑 `/codex:review`
- 质疑"这个做法 / 设计 / 假设对不对" → `/codex:adversarial-review "<关注点>"`
- 改动大或不确定 → 加 `--background`,先去干别的,`/codex:status` 看进度、`/codex:result` 取结果

**红线**:Codex 复审是只读的。拿到结果后停下,按严重度列给用户,问哪些要修——绝不自动改。修不修由用户定。**另外:Codex 只是第一遍、不是证明**——它会漏复杂逻辑 / 罕见路径,说"没问题"≠对,复杂处自己再扫一眼。

## 派活纪律(子代理稳定发挥,主线程耐心等待)
子代理(planner / verifier / Explore / codex)是**全新上下文**——它不知道你们聊过什么。稳定发挥靠简报质量,不靠运气。

**简报五要素**(缺一个就容易跑偏):
1. **背景**:一两句,它需要知道的最小前情;
2. **任务边界**:干什么、干到哪算完;
3. **产出契约**:返回什么形状(清单 / 结论+证据 / PASS·FAIL);
4. **验证 handle**:怎么证明它干对了;
5. **不做什么**:明确禁区(别改文件 / 别越界扫)。

**稳定发挥**:一次派一个明确任务,别塞复合任务;并行时按**不重叠的分片**切;结果含糊或跑偏 → 收紧简报**重派**,别将就着用;子代理的报告也是自我报告——重要结论抽查证据再采信(verifier 的存在就是为这个)。

**耐心等待**(主线程的价值是编排 + 验收,不是抢活):
- 派出去的活**不要自己再做一遍**;也不要结果没回来就当它做完了往下走。
- 会自己通知完成的(子代理、后台任务),安静等通知,**别反复刷状态**——每次空轮询都在烧上下文。
- 只有外部系统(CI / 部署 / 远端队列)值得定期看,间隔跟它真实变化速度匹配。
- 等待期间:有不依赖结果的活就干;没有就明说"在等 X",别没话找话去动代码。

## 执行清单(照着勾)
```
[ ] 分诊:S/M/L/XL 判了级,按级走(S 到此为止,直接改)
[ ] 0 探清:边界 + 相关代码读过了(文件多则派子代理)
[ ] 1 规划:结构/命名定了 + 验证 handle 明确了(多文件走 Plan Mode)
[ ] 2 实现:小步推进,风格一致,没过度设计
[ ] 3 验证:每片跑过,贴了证据
[ ] 3½ (L/XL)checkpoint:next.md 更新到当前进度(做到哪/下一片)
[ ] 4 收拾:/simplify 过、无关改动清掉、无 >300 行巨文件
[ ] 5 复审:/duet:ship(或 /codex:review),findings 交用户定
[ ] 收尾:记下"下一步",收工
```

## Codex 协作速查
| 场景 | 怎么调 | 说明 |
|---|---|---|
| 卡住 / 要第二实现 / 深挖根因 | `/codex:rescue "<明确任务>"` | 可写引擎;小而明确→前台,开放/多步→加 `--background` |
| 整段实现派给 Codex | `/codex:rescue "<完整规格+验证 handle>"` | 先切干净分支/worktree;泡完看 diff + verifier 验收 |
| XL 级并行探查/审计/方案评审 | `/duet:swarm "<任务>"` | 多智能体编排,重火力,小活别用 |
| 收工前一键门禁 | `/duet:ship "<关注点>"` | 收拾 + 验证 + 提示 Codex 复审 + 收尾 |
| 收工前独立查缺陷 | `/codex:review` | 只读、原样返回,不自动修 |
| 质疑设计 / 取舍 / 假设 | `/codex:adversarial-review "<关注点>"` | 挑战 approach 本身,不只挑实现瑕疵 |
| 看后台进度 / 取结果 | `/codex:status` · `/codex:result` | |
| 没装 / 没登录 | `/codex:setup` | 首次准备 Codex |

给 Codex 的 rescue 提示要点:一次一个明确任务;告诉它"做完长什么样";交代验证方式。**codex 必须是 YOLO(approval_policy=never + danger-full-access),否则派活卡在等审批等于没派**——`/duet:init` 会体检并给改法。YOLO 直接改盘,所以护栏是硬的:开跑前先切干净分支 / worktree,泡完先看 diff 再信。安全靠护栏,不靠审批闸。

## 配套习惯(X / 社区验证,顺手就少累)
- **每个逻辑单元提醒用户 commit 一次**(经用户确认后可代提交;不自作主张 commit),别攒到收工:坏决定能单独回滚,最终 diff 干净、好复审——codex YOLO 泡完尤其要先看 diff 再信。
- **上下文到 ~60% 就 `/compact`**,别等撑满:满之前质量就会悄悄下滑,越往后越容易出错、越费你盯。
- **L/XL 任务把 `.duet/next.md` 当活状态文件(长时间工作的断点)**:每完成一片就顺手覆写"做到哪 / 下一片 / 验证 handle"(两三行)。断会话、compact、崩了,随时从 next.md 接上,不靠记忆;收工时再按 ship 覆写成"下一步"。
- **能写测试的地方,先写会失败的测试再实现**——"做完" = 测试变绿,把乱和 bug 挡在源头。

## 收尾
把"下一步 / 明天第一件事"写进项目的 `.duet/next.md`(一两行);**"影响到却没改的其他文件、临时复制没抽共用的地方"也一并记进"未完成"区**,别让跨文件工作跨会话丢。再往 `.duet/journal.md` 追加一条工作日志(日期 / 做了什么 / 怎么验证 / 踩的坑)——项目记忆,下次要背景就来查。下次开会话 duet 的 SessionStart 钩子会自动顶出来,你不用记。然后收工,别把开放循环带过夜——累多半来自事情没关上。
